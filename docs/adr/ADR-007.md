# ADR-007: LightGBM for V2 Model

**Author:** Selvakumar S.
**Created:** 2026-03-12
**Last Updated:** 2026-04-23
**Status:** Accepted

## Context

V1 sklearn RF (AUC 0.8149 — 5-fold cross-val on synthetic data) has validated the CMS compliance pipeline. V2 needs to demonstrate real MLOps: MLflow autologging, Model Registry lifecycle, artifact integrity checks in CI, and a measurable performance story.

## Decision

LightGBM `LGBMClassifier` for V2 with `mlflow.lightgbm.autolog()`. Serialized as portable JSON booster (`denial_model_v2.json`).

## Rationale

LightGBM uses leaf-wise tree growth (splits highest-gain leaf first) rather than level-wise, resulting in higher AUC with the same feature set and training data. MLflow's `lightgbm.autolog()` captures richer artifact detail than sklearn — feature importance plots, booster params, training metrics per iteration — all logged without manual instrumentation. The JSON booster format is human-readable and portable across Python versions and platforms.

`SHAP TreeExplainer` works identically on LightGBM — exact Shapley values, same CMS compliance output, no interface change in `explainability.py` beyond the array indexing branch.

## AUC Context

| Model | AUC | Method | MLflow Stage |
|---|---|---|---|
| V1 sklearn RF | 0.8149 | 5-fold cross-val | Staging |
| V2 LightGBM | 0.8091 | 20% held-out test | Production |

V2's slightly lower number reflects a stricter measurement method (held-out test vs cross-val) — not weaker performance. `train_lgbm_model.py` computes `v1_auc_baseline` using the same cross-val method for a fair comparison, visible in Databricks MLflow experiment runs.

## Alternatives Considered

1. **XGBoost**: similar performance to LightGBM, but binary format is less portable and MLflow autolog captures less detail. JSON serialization requires extra steps.
2. **CatBoost**: good with categorical features but harder to tune, less common in healthcare analytics pipelines, larger model size.
3. **Neural network**: no `TreeExplainer` support — would require approximate SHAP methods, losing the exact attribution guarantee required for CMS defensibility.

## Consequences

- V2 SHAP output is a single 2D array — `explainability.py` branches on `DENIAL_MODEL_VERSION` env var.
- Silver schema is **identical** — downstream dbt, Snowflake, and Gold tables require zero modification when switching between V1 and V2.
- MLflow Model Registry: `denial-risk-sklearn` → Staging, `denial-risk-lgbm` → Production. Aliases used (`staging`, `production`) instead of deprecated stage transitions — requires MLflow ≥ 2.9 (handled by version branch in `mlflow_utils.py`).
- `FORCE_RETRAIN=true` in CI ModelTraining stage ensures both models are always retrained — artifact existence check no longer short-circuits CI.
- **Model Serving (Phase 3 — deferred)**: `infra/modules/databricks/model_serving.tf` is a documented stub. Databricks Model Serving endpoint provisioning is deferred pending Azure VM quota increase for `Standard_D4s_v3` (ADR-007). The serving endpoint would reference `denial-risk-lgbm@production` from the Registry.
