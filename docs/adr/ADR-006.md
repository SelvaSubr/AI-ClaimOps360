# ADR-006: Databricks + Snowflake Hybrid Architecture

**Author:** Selvakumar S
**Created:** 2026-03-12
**Last Updated:** 2026-04-23
**Status:** Accepted

## Context

The pipeline has two distinct workload types: streaming and ML (high PySpark, Delta Lake, MLflow) and analytics (high SQL, concurrent BI users, LLM in SQL). No single platform optimises well for both without significant compromise.

## Decision

**Databricks** for the streaming and ML half: EDI ingestion → Bronze → Silver → AI scoring → Gold Delta tables → MLflow Model Registry.

**Snowflake** for the analytics half: Gold schema → dbt staging/intermediate/marts → Cortex LLM → Snowsight dashboards.

**Data transfer:** Delta Gold → Snowflake via `load_gold_to_snowflake.py` (`COPY INTO` from ADLS Gen2 external stages).

**dbt dual-target:** `dbt_project.yml` defines both `databricks` and `snowflake` profiles. CI Deploy stage runs both targets explicitly.

## Rationale

Databricks is the right tool for Spark workloads, Delta Lake native integration, and MLflow experiment tracking. Snowflake is the right tool for SQL analytics with virtual warehouse isolation (no resource contention between BI queries and dbt runs), concurrent users, and Cortex AI in SQL.

Using both demonstrates right-tool-for-workload engineering — a pattern common in production health plan data platforms — rather than forcing one platform to do everything sub-optimally.

## Alternatives Considered

1. **Pure Databricks** (SQL Warehouse for analytics): weaker SQL analytics performance, no Cortex COMPLETE() in SQL, Databricks SQL Warehouse costs more than Snowflake for concurrent BI workloads.
2. **Pure Snowflake**: no native Spark structured streaming, no MLflow integration, difficult real-time Delta writes from Kafka.
3. **Azure Synapse**: neither strong at Spark streaming nor SQL analytics compared to purpose-built tools. Rejected for lack of MLflow and Cortex equivalents.

## Consequences

- **Dual-platform operations** are independent — Databricks cluster autoterminate (15 min) and Snowflake warehouse autosuspend (10 min) each manage their own costs.
- **dbt dual targets**: Databricks target runs locally only (requires live Unity Catalog tables from streaming jobs); Snowflake target runs in CI via `make snowflake-load` ADLS external table refresh. Both profiles documented in `.env_example`.
- **Data load latency**: `COPY INTO` from ADLS Gen2 → Snowflake adds ~2-5 minute lag between Delta Gold write and Snowflake availability. Acceptable for analytics; for real-time requirements a Kafka Snowflake connector is the upgrade path.
- **Terraform** provisions both platforms — `modules/databricks/` and `modules/snowflake/` — from a single `terraform apply`.
- **Snowflake account identifier** uses org-account format (`ORGNAME-ACCOUNTNAME`) for new Enterprise trial accounts — documented in `infra/modules/snowflake/main.tf` provider block.
