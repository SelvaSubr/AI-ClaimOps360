# ADR-005: SHAP TreeExplainer for CMS-Compliant Feature Attribution

**Author:** Selvakumar S
**Created:** 2026-03-06
**Last Updated:** 2026-04-23
**Status:** Accepted

## Context

The 2024 CMS Non-Discrimination in AI rule requires quantified feature attribution for any AI-supported claim decision. Raw model probabilities and global feature importance do not satisfy this requirement ‚Äî attribution must be per-decision, quantified, and auditable.

## Decision

`SHAP TreeExplainer` for both V1 (sklearn RF) and V2 (LightGBM). Shapley values normalised to percentage contributions stored in `decision_basis` JSON column. 7 CMS-mandatory fields written to Silver on every scored claim.

**7 CMS compliance fields in Silver:**
- `denial_risk_score` ‚Äî integer 0-100, SHAP-normalised
- `ai_decision_only` ‚Äî hardcoded `False` (Requirement 2)
- `human_review_required` ‚Äî `denial_risk_score >= 70` (Requirement 3, rule not model)
- `reconsideration_right` ‚Äî hardcoded `True` (Requirement 4)
- `provider_explanation` ‚Äî SHAP top-feature narrative (Requirement 5)
- `decision_basis` ‚Äî full SHAP attribution JSON (audit trail)
- `model_version` ‚Äî `v1` or `v2` from `DENIAL_MODEL_VERSION` env var

## Rationale

`TreeExplainer` computes **exact** Shapley values (not approximations) for tree models by exploiting tree structure ‚Äî polynomial time rather than exponential. This provides legally defensible, deterministic attribution: the same input always produces the same explanation. This is a hard requirement for CMS audit defensibility.

`ai_decision_only=False` and `reconsideration_right=True` are architectural invariants ‚Äî hardcoded constants, not model outputs. This makes CMS violations structurally impossible regardless of what the model predicts.

## Alternatives Considered

1. **LIME**: approximate, non-deterministic ‚Äî same input produces different explanation on re-run. Legally indefensible for claim denial attribution.
2. **Simple feature importance** (gain/split): global not local ‚Äî does not satisfy the per-decision attribution requirement. Cannot explain why claim X was scored 94 vs claim Y at 12.
3. **Raw probability output**: no attribution at all ‚Äî explicitly rejected by the CMS rule.

## Consequences

- SHAP adds ~50ms per prediction for V2 LightGBM on 6 features ‚Äî acceptable latency for batch scoring.
- V1 and V2 have different SHAP output shapes ‚Äî `explainability.py` branches on model version (V1: `shap_values[1][0]` list-by-class; V2: `shap_values[0]` single 2D array).
- `decision_basis` is a JSON **string** column in Silver ‚Äî not a struct. This is intentional for cross-platform compatibility (Snowflake, dbt, external BI tools). Downstream parsing uses `PARSE_JSON()` in Snowflake SQL.
- `provider_specialty_code` and `days_since_service` default to `11.0` and `30.0` respectively when not present in source data ‚Äî marked as TODO for Phase 3 enrichment from 834 enrollment data.
