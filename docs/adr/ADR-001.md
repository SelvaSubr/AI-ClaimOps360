# ADR-001: Medallion Architecture (Bronze/Silver/Gold)

**Author:** Selvakumar S
**Created:** 2026-03-05
**Last Updated:** 2026-04-23
**Status:** Accepted

## Context

The pipeline processes healthcare claims arriving as raw EDI X12 837P text. Downstream systems need structured, schema-enforced data. Multiple workflows (data engineering, analytics, ML scoring, payer adjudication) need to access data at different quality levels without blocking each other. A bug in any single transform must not require re-ingestion from source.

## Decision

Three-layer Delta Lake architecture:
- **Bronze** — raw, append-only, preserves Kafka offset
- **Silver** — typed, MERGE INTO for idempotent upserts, 7 CMS AI compliance fields
- **Gold** — aggregated, rebuilt from Silver on every run (3 Delta tables + remittance)

## Rationale

Bronze is append-only for replay safety — if the Silver transform has a bug, replay from Bronze without re-pulling from Kafka. Silver is idempotent — running `bronze_to_silver.py` twice produces identical results, which is required for exactly-once semantics. Gold is always derivable from Silver — it is never independently written to, eliminating inconsistency between aggregations and source data.

This also cleanly separates concerns: Bronze is the landing zone, Silver is the contract layer (schema enforcement + AI scoring), Gold is the analytics layer. dbt reads from Gold/Silver and writes to Snowflake — it never touches Bronze.

## Alternatives Considered

1. **Lambda architecture** (batch + speed layers): rejected — two code paths for the same logic, double the operational overhead with no benefit at this scale.
2. **Single-layer** (write processed data directly): rejected — no replay capability, no separation between raw and processed, no schema contract enforcement.
3. **Kappa architecture with streaming-only**: considered for Phase 2 — streaming inference fan-out uses this pattern. Not applied to the full pipeline because batch replay from Bronze is a CMS audit requirement.

## Consequences

- Bronze stores raw FHIR JSON + source EDI metadata — storage is ~2x raw input size.
- Silver has a fixed schema contract — 13 base columns + 7 CMS AI fields. Adding columns requires a `ALTER TABLE` migration.
- Gold queries are always fast (pre-aggregated into `claims_summary`, `provider_summary`, `denial_summary`) and never stale (rebuilt on each pipeline run).
- Remittance data lands in `gold/remittance/` as a separate Delta table — consumed by `remittance_consumer.py` from `remittance.835` Kafka topic.
