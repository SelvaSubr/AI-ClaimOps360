# ADR-003: sklearn RandomForest as V1 Baseline Model

**Author:** Selvakumar S
**Created:** 2026-03-06
**Last Updated:** 2026-04-23
**Status:** Accepted

## Context

Need to validate the CMS compliance pipeline (SHAP, 7 required fields) before building the production model. The project needs a measurable baseline to demonstrate MLOps improvement narrative and model-agnostic compliance.

## Decision

sklearn `RandomForestClassifier` + `StandardScaler` pipeline for V1. Deliberate two-phase approach: V1 validates that the CMS compliance architecture works with any tree model; V2 (LightGBM) demonstrates the full MLOps lifecycle including Registry promotion and hot-swap.

## Rationale

Starting directly with LightGBM eliminates the improvement narrative. The two-phase design is both a portfolio asset and a production pattern: it demonstrates that the CMS compliance architecture is model-agnostic — same SHAP interface, same Silver schema, same 7 CMS output fields regardless of model version. `DENIAL_MODEL_VERSION=v1` hot-swaps back to V1 with zero code changes.

`SHAP TreeExplainer` works identically on both sklearn RandomForest and LightGBM — exact Shapley values, not approximations.

## AUC Numbers

| Model | AUC | Measurement Method |
|---|---|---|
| V1 sklearn RF | **0.8149** | 5-fold cross-validation on full training set |
| V2 LightGBM | **0.8091** | 20% held-out test set |

These are not directly comparable: V1 cross-val AUC is slightly inflated relative to a held-out split. `train_lgbm_model.py` computes `v1_auc_baseline` using the same cross-val method as V1 for a fair side-by-side — this value is captured in Databricks MLflow runs, not in local `mlruns/`.

## Alternatives Considered

1. **Start directly with LightGBM**: rejected — eliminates the improvement narrative and the model-agnostic compliance demonstration.
2. **Logistic Regression**: no `TreeExplainer` support — would require `KernelExplainer` which is approximate and slow (~10s per prediction vs ~50ms for TreeExplainer).
3. **XGBoost for V1**: similar performance to RF but no improvement story; RF is more interpretable as a baseline.

## Consequences

- `explainability.py` branches on `DENIAL_MODEL_VERSION` — V1 uses `shap_values[1][0]` (list-by-class from RF multi-class API), V2 uses `shap_values[0]` (single 2D array from LightGBM).
- Silver schema is **identical** for both versions — no downstream dbt, Snowflake, or Gold changes when switching models.
- `FORCE_RETRAIN=true` in CI ensures both models are always retrained from scratch — `denial_model_v1.pkl` and `denial_model_v2.json` in git are the committed artifacts, overwritten on each CI run.
