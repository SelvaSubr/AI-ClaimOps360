# ADR-004: Snowflake Cortex COMPLETE() for LLM Denial Explanations

**Author:** Selvakumar S
**Created:** 2026-03-06
**Last Updated:** 2026-04-23
**Status:** Accepted

## Context

The Snowflake Gold layer contains claim adjudication data including denial reason codes and SHAP-derived feature attributions. Business users and CMS compliance reviewers need natural language explanations of denial patterns — not raw codes. LLM generation should happen where the data lives, not in an external system, because this is a healthcare application handling PHI.

## Decision

`SNOWFLAKE.CORTEX.COMPLETE('llama3.1-70b', prompt)` called directly in a dbt SQL model (`dbt/models/marts/denial_explanations.sql`) — no external API calls, no data transfer out of Snowflake.

## Rationale

Healthcare data never leaves Snowflake's security perimeter. No API credentials to manage, no data transfer agreements, no Business Associate Agreement (BAA) with an external LLM provider. The entire implementation is two SQL function calls in a dbt model. Cortex runs on Snowflake's managed compute — no separate LLM infrastructure to operate, provision, or scale.

The output is materialized as a table — subsequent `SELECT` statements read stored LLM text without re-invoking Cortex on every query.

## Alternatives Considered

1. **External OpenAI API**: data leaves Snowflake, requires API key management, BAA required for PHI, external latency, cost unpredictable at scale.
2. **Azure OpenAI**: same data perimeter concerns, additional credential management, tighter Azure lock-in.
3. **Local LLM (Ollama)**: not cloud-native, no production scalability, no Snowflake integration, cannot run in dbt SQL.

## Consequences

- **Enterprise edition required** — Cortex is not available on Standard. Verify: `SELECT CURRENT_EDITION();`
- **Region-specific** — `llama3.1-70b` is available in select Snowflake Azure regions. This project uses `east-us.azure`. Production deployments must verify regional availability; fallback to `mixtral-8x7b` if unavailable.
- **Cost**: Cortex COMPLETE is per-token. `denial_explanations.sql` uses `WHERE denial_risk_score >= 50` to limit invocations to high-risk claims only.
- **Run separately**: `dbt run --select denial_explanations --target snowflake` — this model is excluded from default `dbt run` to avoid Cortex charges on every pipeline run.
