# ADR-002: Apache Kafka for Ingestion Decoupling

**Author:** Selvakumar S
**Created:** 2026-03-05
**Last Updated:** 2026-04-23
**Status:** Accepted

## Context

EDI claims arrive at variable rates with burst patterns. Downstream processing (Bronze write, AI scoring, payer adjudication, remittance, archiving) varies in latency. Coupling ingestion directly to any downstream component risks data loss on failure.

## Decision

Apache Kafka (Docker locally via `docker-compose.yml`, Azure Event Hubs in production) as the message bus between EDI parsing and all downstream consumers.

**5 topics:**
- `claims.raw` — FHIR-mapped claim JSON from producer
- `claims.validated` — passed validation
- `claims.rejected` — failed validation
- `claims.review_queue` — high-risk score (≥70), awaiting human review
- `remittance.835` — payer adjudication results

**4 dedicated consumer groups:**
- `claims-bronze` — `claim_consumer.py` → Delta Bronze
- `claims-payer` — `claim_processor.py` → payer adjudication
- `claims-remittance` — `remittance_consumer.py` → Delta Gold/remittance
- `claims-archive` — `transaction_archiver.py` → NDJSON archive

## Rationale

Kafka decouples ingestion rate from processing rate. If any Spark job is down, claims are retained on Kafka for up to 7 days. Four independent consumer groups each maintain their own offset — the Bronze writer, payer processor, remittance consumer, and archiver all read `claims.raw` at their own pace without blocking each other. Replay from any offset is free — critical for reprocessing after a Silver schema migration or CMS audit.

Each pipeline run resets all 5 topics (delete + recreate) to ensure deterministic test runs. In production, this reset is replaced with consumer group offset management.

## Alternatives Considered

1. **Direct Delta writes**: tight coupling — parser must wait for Delta write to succeed, data lost if Spark is down.
2. **Azure Service Bus**: simpler but lacks consumer group semantics and message replay.
3. **Azure Event Grid**: event routing, not streaming — no persistence, no replay.
4. **Single consumer group**: rejected — offset conflicts between Bronze writer and payer processor caused race conditions in early development.

## Consequences

- Adds Docker dependency for local development (`docker/docker-compose.yml`).
- Azure Event Hubs in production (~$10/month base for Standard tier).
- `kafka_offset` column in Bronze enables idempotent replay.
- `broker.address.family: v4` set on all clients — suppresses IPv6 probe noise on macOS.
- Topic names use dots (`claims.raw`) — Kafka warns about period/underscore collisions, but dots are preserved for EDI naming convention alignment.
